NIP-N2
======

Nostr relay discovery, subdomains, and Tor routing via Namecoin
---------------------------------------------------------------

`draft` `optional`

This NIP specifies how a Nostr client opens a `wss://<name>.bit/` (or
`ws://<name>.bit/`) relay connection by resolving the host through
Namecoin and routing it to a real WebSocket endpoint, optionally over
Tor. It builds on NIP-N0 and is independent of NIP-N1 (the same
record may carry both, but neither requires the other).

It is wire-compatible with [ifa-0001][ifa-0001] (`map`, `import`,
`tor`) and [ifa-0002][ifa-0002] (the non-DNS `tor` item type
restated below for the `_tor.txt` shape used in the wild).

[ifa-0001]: https://github.com/namecoin/proposals/blob/master/ifa-0001.md
[ifa-0002]: https://github.com/namecoin/proposals/blob/master/ifa-0002.md

## Abstract

A user lists `wss://example.bit/` (or `wss://relay.example.bit/`) in
their NIP-65 relay set. At connect time, the client:

1. Looks up the parent name (`d/example`).
2. Walks `map` if the host has subdomain labels.
3. Reads `relay` / `relays` (or `nostr.relay` / `nostr.relays`).
4. Reads `_tor.txt` / `tor` for Tor onion endpoints.
5. Picks a wire URL according to the user's clearnet/Tor preference
   and connects.

The `.bit` URL remains the **canonical** relay identity used by event
tags, NIP-65 lists, UI rendering, etc. Only the wire-level handshake
URL is rewritten, in the way a CDN remaps `Host` / SNI.

## Identifier grammar

| Form | Treatment |
|---|---|
| `wss://<name>.bit/` | Top-level lookup on `d/<name>` |
| `ws://<name>.bit/` | Identical to wss but plaintext on the wire |
| `wss://<sub>.<name>.bit/` | Walk `map.<sub>` inside `d/<name>` |
| `wss://a.b.<name>.bit/` | Walk `map.b.map.a` inside `d/<name>` (DNS order) |
| `wss://<name>.bit/<path>` | Path is preserved verbatim from input to wire URL |
| Any non-`.bit` URL | Pass through unchanged — this NIP MUST NOT touch it |

Hosts that are exactly `*.bit` with no second label MUST be rejected.
Hosts with uppercase `.BIT` SHOULD be lowercased before lookup. The
`<name>` part MUST match the ifa-0001 regex.

## Resolution

Read fields from the merged Domain Name Object (i.e. after
`import` resolution and `map` walking; see NIP-N0). Implementations
SHALL try the following keys in order at the walked node, returning
the first non-empty result:

1. `relay` — string; URL.
2. `relays` — array of strings; URLs.
3. `nostr.relay` — string; URL.
4. `nostr.relays` — array of strings; URLs.
5. `nostr.relays[<pubkey>]` — array of strings; URLs. **Last resort**;
   used only when `nostr.relays` itself is keyed by hex pubkey rather
   than being an array. The pubkey MAY be derived from a NIP-05
   lookup of `_@<name>.bit` per NIP-N1 if the publisher chose this
   shape.

Each entry MUST be parsed as a URL per RFC 3986. Entries that fail
parsing or use a scheme other than `ws` / `wss` MUST be dropped.
Duplicates MUST be deduplicated. The order of remaining entries MUST be
preserved as the failover order.

## Subdomain walk

Per NIP-N0 §"Subdomains", the host is split into labels and walked
DNS-order through `map`. The relay/TLSA/Tor fields are read **only**
from the walked node. Ancestors do **not** contribute. This is the
load-bearing security property — without it, a parent record could
silently authorize relay traffic for any subdomain.

Specifically:

- `wss://relay.example.bit/` resolved through `d/example` MUST NOT
  inherit a `relay`, `tls`, `tor`, or `_tor` field declared at the
  root of `d/example`. It uses **only** `value.map.relay.<field>`.
- The `*` wildcard inside `map` MAY be used to delegate any subdomain
  to a default node.
- The `""` empty-key default per ifa-0001 §"map" MAY be used; if both
  exact and `""` exist, exact wins.

A subdomain trap (`d/relay.example` directly registered by an
attacker) MUST NOT be queried as a fallback when `map.relay` is
missing.

## Tor / .onion routing

Two equivalent shapes are accepted:

```jsonc
{ "tor": "x.onion" }                    // ifa-0002 §"tor", string form
{ "tor": [ "x.onion", "ws://y.onion/" ] }   // ifa-0002 §"tor", array

{ "_tor": { "txt": "x.onion" } }            // legacy shape, common in the wild
{ "_tor": { "txt": [ "x.onion", "ws://y.onion/" ] } }
```

Both forms MUST be accepted. Implementations:

- MUST promote bare `.onion` strings to `ws://<onion>/`.
- MUST accept pre-formed `ws[s]://...onion[/path]` URLs verbatim.
- MUST drop entries that are not `.onion` or that use a non-WebSocket
  scheme. (A non-onion under `tor` would otherwise be a record-level
  primitive for redirecting Tor traffic to clearnet.)
- MUST reject multi-label `.onion` hostnames (`foo.deadbeef.onion`)
  so a record cannot publish a fake prefix in front of a real onion.

The full ifa-0002 grammar (string vs empty-array vs array of strings)
is supported. The `i2p` and `freenet` items defined by ifa-0002 are
out of scope for this NIP and SHOULD be ignored unless a future NIP
extends this track.

### Routing decision

Per outgoing connect:

| Record has `tor`/`_tor` | User has Tor on **and** prefers `.onion` for relays | Wire URL |
|---|---|---|
| no  | no  | clearnet `relay`/`relays` |
| no  | yes | clearnet `relay`/`relays` |
| yes | no  | clearnet `relay`/`relays` (loud failure if no clearnet field) |
| yes | yes | onion endpoint, routed via the user's Tor proxy |

A record with **only** an onion (no clearnet `relay`) is supported.
With Tor on, it just works. Without Tor, the connect MUST fail loudly
rather than silently downgrading to a non-existent clearnet endpoint.

The "user prefers `.onion` for relays" toggle is a per-user setting
that this NIP does not standardise; implementations are free to bind
it to whatever Tor UX they already expose. The toggle MUST take
effect per-connect, not just at app start.

## Path preservation

The path component of the input URL (`wss://<host>.bit/<path>`) MUST
be preserved on the wire URL when the resolved entry has no path of
its own. When the resolved entry has a path, the resolved path wins.
This matches the typical relay-mapping deployment where a single
`wss://relay.example.com/inbox` backs `wss://example.bit/`.

## Caching and TTL

A single Namecoin RPC call SHOULD service all of NIP-N1, NIP-N2, and
NIP-N3 for the same host. Implementations SHOULD cache the parsed
Domain Name Object and serve relay-resolution, TLSA, Tor, and identity
queries from the same cache entry.

TTLs as in NIP-N1 §"Caching".

## Identity / relay coupling

If the same host serves both an identity (NIP-N1) and a relay (this
NIP), they SHARE the same Namecoin record. A client that has already
verified `alice@relay.example.bit` MUST NOT need a second ElectrumX
round-trip to learn the relay endpoint, and vice versa.

## Security considerations

- The `.bit` URL is the publisher's *intent*. Rewriting it on the
  wire is a UX convenience, not an authorisation primitive — the
  relay handshake itself MUST still authenticate the resolved host
  (NIP-N3 covers the TLSA-pinning case for the most common
  deployment).
- A malicious resolver could redirect `.bit` traffic to an attacker
  endpoint. Pinning the ElectrumX trust store, validating SPV
  headers, or using a local Namecoin Core RPC are the available
  defences. NIP-N3 mitigates downstream by pinning the relay's TLS
  identity.
- Path preservation means a publisher can deliver per-tenant routing
  through `.bit` URLs. This is intentional. Applications that treat
  the path as a security boundary MUST validate it against the same
  trust anchors as the host.
- Onion-only records are not a privacy guarantee on their own — the
  ElectrumX lookup that produced them MUST itself have been done over
  Tor, otherwise the lookup leaks the user's interest in the host.

## Reference deployment

`d/testls`:

```jsonc
{
  "import": "dd/testls",
  "ip":    "107.152.38.155",
  "_tor":  { "txt": "<old-onion>" }
}
```

`dd/testls`:

```jsonc
{
  "map": {
    "relay": {
      "ip":    "23.158.233.10",
      "relay": "wss://relay.testls.bit/",
      "tls":   [ [ 2, 1, 1, "<HASH>" ] ],
      "_tor":  { "txt": "<new-onion>" }
    }
  },
  "nostr": {
    "names": { "_": "<hex>", "m": "<hex>", "seed": "<hex>" }
  }
}
```

This single deployment exercises:

- NIP-N0: `import`, `map`, `nostr.names`.
- NIP-N1: `m@testls.bit` and `_@testls.bit`.
- NIP-N2: `wss://relay.testls.bit/`, with a Tor fallback.
- NIP-N3: TLSA pinning of the resolved relay handshake.
