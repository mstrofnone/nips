NIP-N0
======

Namecoin record container for Nostr (`nostr` item)
--------------------------------------------------

`draft` `optional`

This NIP defines a single new item type, `nostr`, that lives **inside** a
Namecoin Domain Name Object as defined by [ifa-0001 §"The Domain Name
Object Schema"][ifa-0001]. All Nostr-aware Namecoin features defined in
NIP-N1 (NIP-05), NIP-N2 (relays), and NIP-N3 (TLSA) read fields from this
item and from existing ifa-0001 / ifa-0002 / ifa-0003 items. **No
top-level Namecoin record key is added by any NIP in this track.**

[ifa-0001]: https://github.com/namecoin/proposals/blob/master/ifa-0001.md
[ifa-0002]: https://github.com/namecoin/proposals/blob/master/ifa-0002.md
[ifa-0003]: https://github.com/namecoin/proposals/blob/master/ifa-0003.md

## Motivation

Namecoin's `d/<name>` records (and the parallel `id/<name>` identity
records) carry up to 520 bytes of UTF-8 encoded JSON. The Nostr
ecosystem already has well-defined identifiers (32-byte hex pubkeys,
`wss://` relay URLs, NIP-65 relay lists), so the goal of this track is
to embed Nostr's identifiers in the existing record schema with as
little new surface area as possible.

The `nostr` item is a single JSON object that holds **only** mappings
that have no DNS analogue: hex pubkeys keyed by local-part, and (for
historical reasons) per-pubkey relay hints. Wire-level reachability
(IPs, TLSA, Tor, subdomains) is expressed through the existing
ifa-0001/2/3 items the rest of the .bit ecosystem already understands.

This separation means a Namecoin name can serve a clearnet website, a
.onion, a Nostr identity, **and** a Nostr relay from the same record,
without any of those surfaces having to know about the others.

## Container item

The Domain Name Object MAY contain an item with key `nostr`. Its value
SHALL be a JSON object. Implementations MUST ignore unknown keys inside
the `nostr` object so that this NIP track can be extended in the
future. An object with no recognised keys is equivalent to the absence
of the item.

```jsonc
{
  "nostr": {
    "names":   { "<localpart>": "<32-byte-hex-pubkey>", ... },
    "relays":  { "<32-byte-hex-pubkey>": [ "wss://...", ... ] },
    "relay":   "wss://example.com/",          // OPTIONAL — see NIP-N2
    "relays":  [ "wss://a/", "wss://b/" ]     // OPTIONAL — see NIP-N2
  }
}
```

(`nostr.relay` and `nostr.relays-as-array` are alternative
relay-discovery shapes used by NIP-N2; full grammar and resolution
order is in that NIP.)

The exact shape of `names` and per-pubkey `relays` mirrors NIP-05's
`/.well-known/nostr.json` response so that an implementation can
reuse its existing NIP-05 parser. See NIP-N1 §"Wire format" for
normative details.

### Reserved local-parts

The local-part `_` is RESERVED for the root identity associated with
the name itself, matching NIP-05's `_` convention. Implementations MUST
treat `_@<name>.bit` as a request for `nostr.names._`.

The local-part `*` is RESERVED for future wildcard semantics and MUST
NOT be used by publishers in this track. Implementations MUST NOT
silently substitute `*` for a missing local-part lookup.

Other local-parts are unrestricted other than NIP-05's case-sensitive
matching rules.

### Encoding constraints

- Hex pubkeys MUST be lowercase 64-character hex strings (32 bytes).
  Implementations MUST reject any other shape (e.g. `npub1...`,
  uppercase hex, prefixed `0x...`).
- Relay URLs MUST be valid `ws://` or `wss://` URLs as parsed by
  [RFC 3986][rfc3986]. Implementations MUST drop entries that fail
  parsing or use other schemes.

[rfc3986]: https://www.rfc-editor.org/rfc/rfc3986

## Subdomains

Multi-label hosts (`alice.example.bit`, `relay.example.bit`,
`a.b.example.bit`) are realised through ifa-0001's
[`map`](https://github.com/namecoin/proposals/blob/master/ifa-0001.md#map)
item — they are **not** registered as separate Namecoin names. The
`d/` namespace is single-label.

When NIP-N1 / NIP-N2 walk a subdomain in pursuit of a Nostr-relevant
field, they use the same walker the rest of the ifa-0001 ecosystem
uses. In particular:

1. Split the host on dots, drop the trailing `.bit`, and reverse so
   labels are processed in DNS order (rightmost-most-significant
   first).
2. From the parent record's root, descend `value.map.<label>` for each
   remaining label.
3. At each step, prefer an exact label match. Fall back to
   `value.map["*"]` (wildcard). Honour the `""` empty-key default per
   ifa-0001.
4. Stop at the deepest matching node and read the Nostr-relevant
   fields **only from that node**.

Implementations MUST NOT inherit `nostr`, `relay`, `relays`, `tls`, or
`tor` from ancestor nodes when walking. A subdomain only gets what it
declares (or what comes through the wildcard / empty-key rules).

## `import`

The Namecoin `import` item from
[ifa-0001 §"import"](https://github.com/namecoin/proposals/blob/master/ifa-0001.md#import)
is fully supported. The 520-byte limit makes splitting heavy boilerplate
(`tls`, `_tor`, large `map` blocks) into a sibling name a common
deployment pattern, and the Nostr fields participate in the merge
exactly like every other field.

Implementations:

- MUST resolve `import` before reading any field defined in this track.
- MUST honour importer-precedence: the importing object's items win
  over the imported items, including `null` (which suppresses the
  imported value).
- MUST resolve at least 4 levels of recursion (the spec minimum).
  Deeper chains MAY be silently truncated.
- MUST break cycles via a per-chain visited set.
- SHOULD treat a failed `import` lookup (network error, missing name,
  malformed JSON) as `{}` rather than failing the importing record;
  this matches the "best effort, never throw" convention already
  established in the .bit identity path. A failed import MUST NOT
  cause the importing record's own fields to be ignored.
- MUST accept all four canonical shapes (string, single-element array,
  pair-array with subdomain selector, full array-of-arrays) per
  ifa-0001 §"import".

## Field precedence summary

When more than one field could supply a Nostr-relevant value at a
walked node, the order is:

1. Items declared at the walked node itself.
2. Items merged in via `import` (importer-wins).
3. Items at ancestor nodes — only when explicitly delegated through
   the wildcard / empty-key rules of `map`.

There is **no** silent fallback from one field to another (e.g. `tls`
on the parent does NOT cover a missing `tls` on a child). NIP-N2 and
NIP-N3 reinforce this property because it is load-bearing for security.

## Security considerations

- A name's owner controls the value, but not what consumers cache. To
  bound poisoning windows, implementations SHOULD use TTLs on the order
  of an hour and SHOULD invalidate cache entries when their owning
  name expires (see NIP-N1 §"Expiry").
- A subdomain trap (`d/<sub>.<parent>` directly registered by a
  squatter) MUST NOT be queried as a fallback when walking
  `map`. Implementations that use ElectrumX or another lookup
  primitive MUST always go through the registered name, never through
  a synthesised one.
- Records using `import` can pull fields from any name in the chain.
  Implementations SHOULD apply the same caching and TTL policy to the
  imported names as to the importing one.

## Backwards compatibility

`nostr` is a previously-unallocated key in the ifa-0001 schema. Existing
ifa-0001 implementations ignore unknown keys per
[ifa-0001 §"Item Suppression Rules"](https://github.com/namecoin/proposals/blob/master/ifa-0001.md#item-suppression-rules),
so adding a `nostr` item to a record does not affect DNS, TLS, or Tor
clients of the same name.
