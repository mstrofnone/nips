NIP-N4
======

Service Attestations with optional Namecoin-anchored identity binding
---------------------------------------------------------------------

`draft` `optional`

This NIP defines two event kinds for **signed attestations of service or
task completion** between Nostr identities, forming a portable, queryable
reputation layer. It also specifies an optional **Namecoin identity
anchor** mechanism, whereby the Nostr pubkey receiving an attestation
MUST have been the active value associated with a Namecoin name at the
time the attested service occurred, providing a blockchain-timestamped
binding between human-readable names and reputation history.

It builds on:

- NIP-N0 (the `nostr` item, `nostr.names`, `import`, `map` walker,
  encoding constraints, security model).
- NIP-N1 (NIP-05 verification via Namecoin) — re-used here as the
  *temporal* lookup primitive (lookup at a specified block height
  instead of "head").
- [NIP-32][nip-32] for label vocabulary.
- [NIP-58][nip-58], [NIP-85][nip-85], [NIP-39][nip-39] as adjacent
  reputation surfaces (see §"Relationship to existing NIPs").
- [ifa-0001][ifa-0001] for the underlying Namecoin record shape.

## Abstract

A `kind:38383` event represents a single attestation by an *attester*
that an *attestee* (identified by hex pubkey) performed a specific
service or task. A `kind:38384` event lets the attester maintain a
curated set of attestations they currently endorse.

If an attestation includes an `nmc` tag, the attester additionally
claims that, at the moment the service was completed, the attestee's
pubkey was the active value of `nostr.names._` (or
`nostr.names.<localpart>`) under a specified Namecoin name. A verifier
re-runs the NIP-N1 lookup *at the cited block height* and confirms the
match.

Attestations without an `nmc` tag are fully valid and useful; the
Namecoin layer is purely additive.

## Motivation

Existing NIPs cover adjacent surfaces but none cover bilateral,
structured, verifiable attestations of services or tasks:

| Existing NIP | What it covers | Gap |
|---|---|---|
| NIP-58 (Badges) | Issuer-defined badges awarded to pubkeys | Binary; no rating, service context, or temporal identity binding. |
| NIP-32 (Labeling) | Generic labels on events / pubkeys / relays | No rating scale, service context, or transaction concept. |
| NIP-56 (Reporting) | Negative reports on content / users | One-directional, negative-only. |
| NIP-85 (Trusted Assertions) | WoT score computation by service providers | Aggregation layer, not first-party attestation events. Complementary; sits *above* this NIP. |
| NIP-39 (External Identities) | Linking platform accounts | Proves account ownership; carries no reputation or attestation data. |

This NIP fills the gap and, optionally, anchors the identity side of
each attestation in Namecoin.

## Specification

### Event kinds

| Kind | Name | Description |
|---|---|---|
| `38383` | Service attestation | Parameterised replaceable. A signed attestation that a pubkey performed a service. |
| `38384` | Attestation set | Parameterised replaceable. The attester's curated list of currently-endorsed attestations. |

> Kind numbers are provisional pending allocation by `nostr-protocol/nips`.

### Kind `38383`: service attestation

A *parameterised replaceable* event published by the attester about the
attestee. The `d` tag makes the event replaceable, allowing the attester
to update their rating without re-publishing the entire history.

```json
{
  "kind": 38383,
  "pubkey": "<attester pubkey (hex)>",
  "created_at": <unix timestamp>,
  "tags": [
    ["p", "<attestee pubkey (hex)>", "<recommended relay URL>"],
    ["d", "<unique attestation identifier>"],
    ["service", "<category>", "<optional subcategory>"],
    ["rating", "<score>", "<max>"],
    ["completed_at", "<unix timestamp of service completion>"],
    ["ref", "<event id, URL, or external identifier>"],
    ["nmc", "<namecoin name>", "<block height at service time>"],
    ["L", "nip-N4.service"],
    ["l", "<service category>", "nip-N4.service"]
  ],
  "content": "<optional free-text review or context>",
  "sig": "<signature>"
}
```

#### Required tags

- **`p`** — Hex pubkey of the attestee. A relay hint is RECOMMENDED per
  [NIP-01][nip-01].
- **`d`** — Unique attestation identifier chosen by the attester. Makes
  the event replaceable so the attester can update the rating later.
- **`service`** — Lowercase, dot-separated category string. The first
  value is the primary category; an optional second value is a
  subcategory. See §"Service category registry" for the initial
  vocabulary.
- **`rating`** — Two-element value: score and maximum. Both MUST be
  non-negative numbers. Clients SHOULD normalise to a 0–1 scale for
  aggregation.
  - `["rating", "4.5", "5"]` → 0.9
  - `["rating", "92", "100"]` → 0.92
  - `["rating", "1", "1"]` → binary positive

#### Optional tags

- **`completed_at`** — Unix timestamp at which the service completed.
  If absent, `created_at` is used. Useful when the attestation is
  published some time after the work.
- **`ref`** — Reference to the underlying transaction. MAY be:
  - An `e` tag (Nostr event ID) for on-Nostr-negotiated work.
  - A URL for an external reference.
  - An NIP-39-style `i` tag for external content IDs.
- **`nmc`** — Namecoin anchor (see §"Namecoin identity anchoring").
- **`L`** / **`l`** — [NIP-32][nip-32] namespace labels. RECOMMENDED for
  discoverability via relay label queries.

#### Content

`content` is an optional free-text review. MAY be empty. Clients SHOULD
render it alongside the structured rating data.

### Kind `38384`: attestation set

A parameterised replaceable event letting the attester maintain a
current-endorsements list. Opinions change; a five-star rating today may
not reflect the attester's view a year later. The set lets the attester
withdraw an old attestation without deleting its event.

```json
{
  "kind": 38384,
  "pubkey": "<attester pubkey (hex)>",
  "created_at": <unix timestamp>,
  "tags": [
    ["d", "active_attestations"],
    ["a", "38383:<attestee pubkey>:<d-tag>", "<relay hint>"],
    ["a", "38383:<attestee pubkey>:<d-tag>", "<relay hint>"]
  ],
  "content": "",
  "sig": "<signature>"
}
```

The `a` tags reference the attester's own kind-38383 events. Clients
MAY treat *only* attestations present in the most recent attestation
set as the attester's currently active endorsements; attestations that
exist but are absent from the set SHOULD be treated as withdrawn.

If an attester has never published an attestation set, all of their
kind-38383 events SHOULD be treated as active.

## Namecoin identity anchoring

### Overview

The `nmc` tag binds an attestation to a Namecoin name at a specific
block height. A verifier reproduces the NIP-N1 lookup *at that height*
and confirms that the attestee pubkey matches the value of
`nostr.names._` (for `<name>.bit` / `id/<name>` form) or
`nostr.names.<localpart>` (for `<localpart>@<name>.bit` form) under the
named record.

This re-uses NIP-N0's container and NIP-N1's lookup; nothing
attestation-specific is added to the Namecoin record itself.

### Record shape

The attestee's identity record uses the same `nostr` item defined in
NIP-N0. **No new top-level Namecoin record key is added by this NIP.**
A minimal record:

```json
{ "nostr": { "names": { "_": "<attestee pubkey hex>" } } }
```

A keyed local-part:

```json
{ "nostr": { "names": { "alice": "<alice pubkey hex>" } } }
```

Multi-key delegation MAY be expressed via additional local-parts (e.g.
`nostr.names.signing`) — these are normal NIP-N0 entries and have no
special meaning here.

### The `nmc` tag

```
["nmc", "<name>", "<block height>"]
```

- **`<name>`** — The full Namecoin name including namespace prefix.
  RECOMMENDED forms:
  - `id/alice` for identity-namespace anchoring.
  - `d/alice` for domain-namespace anchoring (when the same name is
    also serving NIP-N1 / NIP-N2).
  - The `.bit` form (`alice.bit` / `alice@example.bit`) is also
    accepted; verifiers MUST normalise per NIP-N1 §"Identifier grammar"
    before lookup.
- **`<block height>`** — Namecoin block height at, or nearest before,
  the `completed_at` timestamp. Acts as the temporal checkpoint for
  verification.

### Verification procedure

A verifier MUST follow these steps:

1. Read the attestee pubkey from the `p` tag.
2. Read `completed_at` (or fall back to `created_at`).
3. Parse `nmc` to obtain the Namecoin name and reference height **H**.
4. Resolve the name *at height H* using the NIP-N1 lookup primitive.
   The temporal extension to NIP-N1 is:
   - When using ElectrumX, call `blockchain.name.get_value_proof` with
     the block height **H** as the reference and walk the proof to the
     `name_update` that was active at height H.
   - When using Namecoin RPC directly, call
     `name_history "<name>"` and select the entry whose `height` is
     the largest value **≤ H**.
   The result is the merged ifa-0001 Domain Name Object that was
   active at height H, including any `import` resolution per NIP-N0
   §"`import`".
5. Read `nostr.names.<key>` from the resolved record, where `<key>`
   is the local-part of the identifier (or `_` for `<name>.bit` /
   `id/<name>` / `d/<name>` form).
6. Compare. The hex string MUST equal the `p`-tag pubkey.
7. Validate temporal bounds:
   - The block at height H MUST have a timestamp **≤** the
     attestation's `completed_at`.
   - The name MUST NOT have been expired at height H. Per ifa-0001,
     `id/` and `d/` names expire 36 000 blocks after their last
     `name_firstupdate` or `name_update`; verifiers MUST NOT accept an
     `nmc` reference that falls inside an expired interval.
8. Result. If every check passes, the attestation is
   *Namecoin-verified*. Clients MAY render a distinct visual indicator.

### Temporal-gap tolerance

Namecoin blocks occur ≈ every 10 minutes. Name updates require a
12-block confirmation depth. Verifiers SHOULD allow a tolerance window
of **±12 blocks** (~2 hours) around the cited height when matching it
against `completed_at`.

### Key rotation

When an `id/` or `d/` name later updates its `nostr.names._` to a new
pubkey, attestations referencing the previous key remain valid *iff*
their `nmc` height falls inside the previous key's active interval.
Namecoin preserves the full update history, so historical attestations
are always re-verifiable.

Clients SHOULD indicate when an attestation references a pubkey that
is no longer the *current* value, e.g. *"verified at time of service
(key since rotated)"*.

### Why anchor at all

Nostr pubkeys are pseudonymous and cheap to generate. When reputation
has economic value (a contractor's track record, a marketplace
seller's history) there is an incentive to fabricate attestation
histories on a fresh key and *then* associate that key with a public
identity.

Namecoin's `id/` and `d/` namespaces provide a tamper-evident,
temporally-ordered binding between human-readable names and pubkeys.
Because the chain is merge-mined with Bitcoin, a verifier can confirm
that a given pubkey *was* the declared identity of a name at the time
a service was attested. This rules out backdating reputation onto a
name that didn't hold the key at the relevant time.

This is opt-in. Anchorless attestations remain useful. Anchoring adds
a higher tier of assurance for marketplaces, professional services,
and credentialing.

## Querying and aggregation

### Relay filters

Recent attestations involving a specific attestee:

```json
{ "kinds": [38383], "#p": ["<attestee pubkey>"] }
```

Recent attestations of a specific service category (when `L`/`l` are
populated):

```json
{ "kinds": [38383], "#l": ["software.frontend"] }
```

Currently-active attestations by a given attester:

```json
{ "kinds": [38384], "authors": ["<attester pubkey>"], "#d": ["active_attestations"] }
```

### Client-side aggregation

When computing a pubkey's score, clients SHOULD:

- Filter attestations through their NIP-02 web-of-trust before
  weighting.
- Treat Namecoin-verified attestations as a separate, higher-confidence
  tier rather than mixing them with anchorless ones.
- Weight more recent `completed_at` more heavily; old work depreciates.
- Discount attestations whose attester has issued anomalously many
  attestations in a short window (likely sybil cluster).

### NIP-85 integration

NIP-85 trusted-assertion providers MAY ingest kind-38383 / kind-38384
events as input to their `kind:30382` / `kind:30383` outputs. The `nmc`
tag (when present) is a strong signal that should be propagated into
provider-issued assertions.

## Security considerations

### Sybil attacks

- WoT filtering is the primary defence. Attestations from unknown
  pubkeys carry no weight by default.
- Namecoin anchoring raises the cost per identity (each name requires
  NMC plus periodic renewal).
- [NIP-13][nip-13] proof-of-work MAY optionally be required by relays
  or weighted by clients.

### Collusion

- Graph analysis by NIP-85 providers can detect mutual-attestation
  cliques with no outside connections.
- [NIP-57][nip-57] zap receipts referenced via the `ref` tag provide
  evidence of an actual economic transaction underlying the
  attestation.

### Namecoin reorg

A deep Namecoin reorganisation could in principle invalidate a
historical name-to-pubkey binding. In practice, Namecoin is merge-mined
with Bitcoin and has not experienced a meaningful reorg in years.
Clients SHOULD treat `nmc` references whose block height is older than
**100 blocks** as having negligible reorg risk; references inside the
last 100 blocks SHOULD be treated as provisional. (NIP-N0 §"Security
considerations" and NIP-N1 §"Caching" cite the same threshold.)

### Stale Namecoin names

A name that has expired (no `name_update` within 36 000 blocks) loses
its binding. Verifiers MUST reject `nmc` references whose height falls
inside an expired interval.

### Cache poisoning

The general caching guidance in NIP-N0 §"Security considerations" and
NIP-N1 §"Caching" applies. A `nmc`-bound verification result MAY be
cached against `(name, height)` rather than `(name, head)`, which makes
it considerably more stable than a NIP-N1 cache entry.

## Examples

### Example 1: simple attestation, no Namecoin

```json
{
  "kind": 38383,
  "pubkey": "aaa111...",
  "created_at": 1711000000,
  "tags": [
    ["p", "bbb222...", "wss://relay.example.com"],
    ["d", "alice-bob-plumbing-2026-03"],
    ["service", "plumbing.repair"],
    ["rating", "5", "5"],
    ["completed_at", "1710900000"],
    ["L", "nip-N4.service"],
    ["l", "plumbing.repair", "nip-N4.service"]
  ],
  "content": "Fixed the kitchen sink leak quickly and cleanly. Would hire again.",
  "sig": "..."
}
```

### Example 2: Namecoin-anchored attestation

Charlie attests that `id/dana` performed excellent frontend work. At
block 849 500, `id/dana` resolved (per NIP-N1) to
`nostr.names._ = "ddd444..."`.

```json
{
  "kind": 38383,
  "pubkey": "ccc333...",
  "created_at": 1711000000,
  "tags": [
    ["p", "ddd444...", "wss://relay.example.com"],
    ["d", "charlie-dana-frontend-2026-q1"],
    ["service", "software.frontend"],
    ["rating", "95", "100"],
    ["completed_at", "1710500000"],
    ["ref", "https://github.com/project/pull/42"],
    ["nmc", "id/dana", "849500"],
    ["L", "nip-N4.service"],
    ["l", "software.frontend", "nip-N4.service"]
  ],
  "content": "Delivered the React dashboard ahead of schedule. Clean code, great communication.",
  "sig": "..."
}
```

Verification: client resolves `id/dana` at height 849 500 (NIP-N1
temporal lookup), reads `nostr.names._ = "ddd444..."`, confirms it
matches the `p` tag, and marks the attestation Namecoin-verified.

### Example 3: attestation set

```json
{
  "kind": 38384,
  "pubkey": "aaa111...",
  "created_at": 1711100000,
  "tags": [
    ["d", "active_attestations"],
    ["a", "38383:bbb222...:alice-bob-plumbing-2026-03", "wss://relay.example.com"],
    ["a", "38383:eee555...:alice-eve-tutoring-2025", "wss://relay.example.com"]
  ],
  "content": "",
  "sig": "..."
}
```

Alice had previously attested to a third party but has removed that
`a` tag — withdrawing her current endorsement without removing the
underlying event.

## Backwards compatibility

This NIP introduces new event kinds and new tags. Clients and relays
that do not implement it ignore them per [NIP-01][nip-01].

The `nmc` tag is purely additive: an attestation without one is fully
valid; with one, it gains a verifiable identity-time binding.

NIP-32 `L`/`l` tags are included for discovery compatibility. NIP-85
integration is purely additive.

## Relationship to other NIPs in this track

- **NIP-N0** provides the `nostr` item, `nostr.names`, `import`, and
  `map` resolution that the verification procedure consumes.
- **NIP-N1** provides the lookup transport (ElectrumX TCP+TLS or
  WebSocket; Namecoin RPC). NIP-N4 calls it with a block-height
  parameter instead of "head".
- **NIP-N2** and **NIP-N3** are unrelated to attestations but MAY be
  used together: a `.bit` relay can host attestation events and pin
  its own TLS, with attestations on it referencing `id/` or `d/`
  names verified at past block heights.
