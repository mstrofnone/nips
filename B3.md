NIP-B3
======

TLSA pinning for `.bit` Nostr relay WebSockets (RFC 6698)
---------------------------------------------------------

`draft` `optional`

This NIP specifies how a Nostr client pins the TLS handshake of a
`wss://<name>.bit/` (or `wss://<sub>.<name>.bit/`) relay against
[RFC 6698][rfc6698] (DNS-Based Authentication of Named Entities)
records published in the same Namecoin name. The goal is to make a
public-CA MITM ineffective against `.bit` relay traffic even when the
attacker has a valid clearnet certificate for the resolved hostname.

It builds on NIP-B0 and NIP-B2. NIP-B2 produces the wire URL; this
NIP governs the certificate validation that follows.

It is wire-compatible with [ifa-0001][ifa-0001] §"tls" and
[ifa-0003][ifa-0003] (Dehydrated TLS Certificates), without requiring
either side of the connection to dehydrate or rehydrate certificates.

[rfc6698]: https://www.rfc-editor.org/rfc/rfc6698
[ifa-0001]: https://github.com/namecoin/proposals/blob/master/ifa-0001.md
[ifa-0003]: https://github.com/namecoin/proposals/blob/master/ifa-0003.md

## Abstract

A Namecoin name MAY publish one or more TLSA records under the `tls`
item of its Domain Name Object (or, per ifa-0003, under the `d8`
encoding of the same item). When a Nostr client opens
`wss://<name>.bit/` and finds `tls` at the walked node, **at least one
record MUST match** for the connection to proceed. There is no silent
fallback to plain PKIX once `tls` is published.

## Wire format

`tls` is a JSON array of records. Each record is an array of four
items:

```jsonc
"tls": [
  [ <usage>, <selector>, <matchingType>, <association data> ]
]
```

The four-tuple matches RFC 6698:

| Usage | Name | Behaviour |
|---|---|---|
| 0 | PKIX-TA | Trust anchor in chain. Plus full PKIX validation against system trust store. |
| 1 | PKIX-EE | End-entity cert. Plus full PKIX validation. |
| 2 | DANE-TA | Trust anchor in chain. **PKIX bypassed.** |
| 3 | DANE-EE | End-entity cert. **PKIX bypassed.** |

| Selector | Name | Hashes |
|---|---|---|
| 0 | Cert | Full DER cert |
| 1 | SPKI | SubjectPublicKeyInfo |

| MatchingType | Name |
|---|---|
| 0 | Full (raw bytes, no hash) |
| 1 | SHA-256 |
| 2 | SHA-512 |

`<association data>` is a base64-encoded byte string. Implementations
MUST accept both standard and URL-safe base64, with or without
padding, and MUST tolerate whitespace inside the value (Namecoin
records frequently embed line breaks). `kotlin.io.encoding.Base64` and
the JS `Buffer` class do **not** by default — strip whitespace before
decoding.

The legacy [ifa-0003][ifa-0003] dehydrated form is also valid:

```jsonc
"tls": { "d8": [ <ver>, <pubKeyB64>, <notBefore>, <notAfter>, <sigAlg>, <sigB64> ] }
```

Implementations that already speak ifa-0003 MAY accept this; the
straight TLSA form (above) is RECOMMENDED for new deployments because
it does not require certificate rehydration.

## When to validate

Implementations MUST evaluate `tls` only at the walked node (NIP-B0
§"Subdomains"). A `tls` field on `d/<name>` does **not** authorise
the TLS handshake for `wss://<sub>.<name>.bit/`. Each subdomain
publishes its own `tls`, or delegates via `*` / `""` per `map`
semantics.

The `tls` field is consulted **after** the wire URL is resolved by
NIP-B2. The validated host is the resolved host, not the `.bit` host
the user typed. SNI MAY be set to either; the certificate's
SubjectAltName MUST match the resolved host to satisfy classic PKIX
validation in the PKIX-* usage modes.

## Validation algorithm

For each TLS handshake on a `wss://<name>.bit/`-resolved connection:

1. Look up the merged Domain Name Object for the name + walked
   subdomain (NIP-B0).
2. If `tls` is absent, fall through to whatever TLS validation policy
   the implementation uses for non-`.bit` hosts (typically the system
   trust store).
3. If `tls` is present:
   1. Decode each record. Drop records with unknown
      usage/selector/matchingType triples.
   2. For each record, compute the candidate digest from the relevant
      certificate in the presented chain (end-entity for EE records,
      any cert in the chain for TA records) and compare against
      `<association data>`.
   3. **Accept** the handshake iff at least one record matches **and**
      the chain satisfies that record's PKIX requirement (system
      trust store for PKIX-*; bypassed for DANE-*).
   4. **Reject** otherwise. There is no silent fallback to plain
      PKIX. A publisher who shipped TLSA records is opting in to
      mandatory TLSA validation.

The reject MUST surface as a TLS handshake failure at the connection
layer, not as a silent reconnect or a downgrade. The `.bit` relay URL
SHOULD remain in the relay list and MAY be retried later (the
publisher might fix their record); it MUST NOT be quietly demoted to
plain PKIX.

## Multiple records

A name MAY publish multiple TLSA records. The handshake succeeds on
the first match. Records SHOULD be ordered with the most secure /
most desired first; implementations MAY short-circuit on the first
match for performance.

This is the standard rotation pattern: publish both old-key and
new-key TLSA, deploy the new cert, then drop old-key.

## Trust-anchor edge case (Namecoin "AIA-stapled SPKI")

Some `.bit` deployments stash the trust-anchor SPKI inside the leaf
cert's issuer field (encoded as JSON in the issuer's `serialNumber`
RDN), per the
[encaya](https://github.com/namecoin/encaya) /
[ncgencert](https://github.com/namecoin/ncdns) ecosystem. In this
case the on-chain TLSA records are computed against that **stapled**
SPKI, not against the leaf or the chain certs.

Implementations that want to support this deployment SHOULD:

1. Inspect the leaf cert's `issuer.serialNumber` field. If it parses
   as JSON of the form `{"pubb64":"<b64url>"}`, base64url-decode the
   value and treat it as an SPKI for matching purposes.
2. Compute the TLSA digest from this SPKI for usage 2 / selector 1
   records (DANE-TA / SPKI).
3. Match against the `tls` array.

Implementations that don't support this MAY simply not match, in which
case the handshake fails per §"Validation algorithm". This is
acceptable — the publisher can switch to a publicly-rooted cert and
DANE-EE records.

(See `nmcLightningService/publish-announce-bit.js` for the canonical
parser; the cert literally encodes `\"` as backslash+quote, so the
implementation must unescape before `JSON.parse`.)

## Caching

The validated certificate fingerprint SHOULD be cached alongside the
relay-resolution entry (NIP-B2 §"Caching and TTL"). On subsequent
connects, an implementation MAY skip the TLSA recomputation if the
presented certificate's fingerprint already matches the cached one
**and** the cache entry's name is still within its on-chain expiry.

The cache MUST be invalidated when the on-chain `tls` field changes
or the name expires.

## Multi-label hosts

A `tls` field on `d/<name>` does not cover any subdomain. A walked
node at `value.map.relay` (for `relay.<name>.bit`) MUST publish its
own `tls`. The `*` wildcard in `map` MAY be used to share TLSA records
across an unbounded number of subdomains:

```jsonc
{
  "map": {
    "*": {
      "tls": [ [ 2, 1, 1, "<wildcard-hash>" ] ]
    }
  }
}
```

This is the only mechanism by which TLSA records "inherit" — and it
is opt-in and explicit on the publisher's side.

## Backwards compatibility

`.bit` relays whose Namecoin record does **not** carry `tls` are
unaffected: TLS validation falls back to whatever the implementation
already does. This NIP only changes behaviour for relays that opt in
by publishing TLSA records.

Existing TLSA-capable tooling
([`encaya`](https://github.com/namecoin/encaya),
[`ncdns`](https://github.com/namecoin/ncdns), the proposed
[ifa-0003 dehydrated certificates][ifa-0003]) reads the same `tls`
field. Records published per this NIP are simultaneously usable by
DNS-over-Namecoin / TLS-over-Namecoin clients.

## Security considerations

- A `tls` record locks the TLS identity to whatever the publisher
  put on chain. Mistakes (e.g. publishing a hash of the wrong cert)
  break the relay until the next `NAME_UPDATE` confirms — typically
  a few minutes to an hour. Publishers SHOULD validate their TLSA
  records before broadcasting.
- DANE-TA / DANE-EE bypass the system trust store. This is the
  point. It also means the implementation MUST NOT short-circuit on
  "untrusted issuer" before consulting `tls`.
- A publisher that uses DANE-TA (usage 2) is trusting the entire path
  from trust anchor down to leaf, including any intermediate the CA
  rotates. SPKI hashing (selector 1) of the leaf with usage 3
  (DANE-EE) is the tightest binding and SHOULD be preferred for
  long-lived relays.
- Implementations MUST NOT log raw association data with PII; the
  hash itself is non-secret, but the surrounding cert often contains
  organisation names, host SANs, etc.
- The `i2p` and `freenet` items from ifa-0002 are not covered by
  this NIP. They have no TLS layer to pin.

## Reference implementation

The canonical pure-Kotlin implementation is `TlsaVerifier` in Quartz
([`nip05DnsIdentifiers/namecoin`](https://github.com/vitorpamplona/amethyst/tree/main/quartz/src/commonMain/kotlin/com/vitorpamplona/quartz/nip05DnsIdentifiers/namecoin)).
The Android trust-manager wrapper (`TlsaTrustManager`) and OkHttp
overlay (`TlsaConnectionPolicy`) are in the Amethyst app module
(per [PR #2595](https://github.com/vitorpamplona/amethyst/pull/2595)).
The same Quartz code is reused on iOS and Desktop in Amethyst, and
mirrored functionally in Nostur (Swift / NWConnection +
`sec_protocol_options_set_verify_block`) and in `nostr-tools` (TS,
Node `tls` module).
